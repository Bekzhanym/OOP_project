## 1. С чего начать защиту (первые 2 минуты)

### Шаг 1 — представьтесь и назовите цель

Скажите буквально одну фразу:

> Мы сделали **модель университета на Java**: пользователи (студенты, преподаватели и др.), курсы с несколькими преподавателями, учёт научных статей у «исследователей», сохранение состояния в файл и небольшой **консольный демо-сценарий**, который это всё показывает.

### Шаг 2 — покажите структуру «сверху вниз»

Не лезьте сразу в классы. Нарисуйте на доске или скажите словами:

1. **Домен** — «кто есть и какие у них данные» (студент, курс, статья…).
2. **Приложение (application)** — «что система умеет делать» — отдельные сценарии (use case).
3. **Инфраструктура** — «где это хранится» — сейчас это память + файл `data/university-state.ser`.

Скажите: сначала можно подгрузить старый дамп (**y/n**), в конце — сохранить (**y/n**).

## 3. Архитектура проекта (простыми словами)

| Слой | Папка в коде | Зачем нужен |
|------|----------------|-------------|
| **Домен** | `src/kbtu_oop_project/domain/...` | Сущности и правила предметной области: пользователь, курс, статья, исключения, интерфейсы репозиториев. Без «файлов» и консоли. |
| **Приложение** | `application/` | Фабрика пользователей и **use case** — отдельные классы под задачи (поиск, каталог исследователей, регистрация на курс). |
| **Инфраструктура** | `infrastructure/persistence/` | Реализация хранилища: singleton `UniversityDatabase`, сериализация в файл. |
| **Точка сборки** | `UniversityApp.java` | Одно место, откуда консоль берёт базу и готовые use case — меньше связности в `main`. |
| **Демо** | `console/ConsoleDemo.java` | Интерактивный прогон для защиты. |

Фраза для экзамена:

> Домен не знает про консоль и файл. Консоль вызывает сценарии и базу через `UniversityApp`.

## 4. Фичи по отдельности (что говорить про каждую)

Ниже — **каждая тема отдельным блоком**: «что это», «зачем», «где смотреть».

---

### Фича A — Пользователи и роли

**Что это.** В системе есть разные типы людей: студент, преподаватель, сотрудник, админ, менеджер, научный стафф. Общее поведение (логин/пароль, профиль) вынесено наверх.

**Зачем.** Это классическая **иерархия наследования** и полиморфизм: где-то мы работаем с абстрактным `User`, где-то — с конкретным `Student` или `Teacher`.

**Где в коде.**

- Базовый класс и общие поля: `domain/features/user/User.java`.
- Наследники: `Student`, `Teacher`, `Employee`, … в том же пакете `features/user`.
- Роли как перечисление: `domain/value/Role.java`.
- Создание «правильного» класса по роли — **фабрика**: `application/factory/UserFactory.java` (`switch` по `Role`).

**Что сказать одной строкой.**


### Фича B — Студент и учёба (курс, кредиты, транскрипт)

**Что это.** Студент может **записаться на курс** с проверкой лимита кредитов; у него есть внутренний **транскрипт** (оценки по кодам курсов).

**Зачем.** Показать инкапсуляцию и простые инварианты (если правило нарушено — исключение).

**Где в коде.**

- Логика студента: `domain/features/user/Student.java` (`registerForCourse`, поля кредитов, `Transcript`).
- Оценка как объект: `domain/features/course/Mark.java`.
- Транскрипт: `domain/features/course/Transcript.java`.

**Демо.** В `ConsoleDemo` после создания студента и курса вызывается `student.registerForCourse(course)` и выводится сумма кредитов.

---

### Фича C — Курс и несколько преподавателей

**Что это.** У одного курса может быть **несколько преподавателей** (список `Teacher`). При добавлении преподавателя курс и преподаватель **согласуют связь**: у преподавателя должен появиться этот курс в списке ведёмых.

**Зачем.** Показать двустороннюю связь и аккуратный публичный API (`attachTeachingAssignment` / `detachTeachingAssignment`), чтобы разные слои не лезли во «внутренности» другого класса.

**Где в коде.**

- Курс: `domain/features/course/Course.java` — `addInstructor`, `removeInstructor`, список инструкторов.
- Преподаватель: `domain/features/user/Teacher.java` — связь с `Course`.

**Фраза.**


### Фича D — Наблюдатель за курсом (Observer / Subject)

**Что это.** Курс реализует роль **«субъекта»**: на него можно **подписаться** как наблюдатель. При изменении/уведомлении курс рассылает **уведомления** подписчикам.

**Зачем.** Паттерн **Observer**: слабая связность — курс не обязан знать конкретный класс подписчика, только интерфейс.

**Где в коде.**

- Интерфейсы: `domain/features/notification/Observer.java`, уведомление как объект — `Notification.java`.
- Контракт «подписаться / отписаться / уведомить» для курса: `domain/features/course/Subject.java`.
- Реализация на курсе: `Course.java` (`subscribe`, `notifyObservers`, …).
- Базовый пользователь уже умеет принимать уведомления через `update(...)`: `User.java`.

**На защите.** Не обязательно жать всё в демо — достаточно показать код подписки/уведомления и объяснить: «пользователь как наблюдатель».

---

### Фича E — Исследования: интерфейс Researcher и статьи

**Что это.** И студент, и преподаватель (и др.) могут выступать как **исследователи**: у них есть **h-index**, список **ResearchPaper**, методы добавления и печати статей с ** Comparator**.

**Зачем.**

- **Интерфейс `Researcher`** — полиморфизм: один код работает и с преподавателем, и со студентом как с исследователем.
- **`ResearchPaper`** — отдельная сущность с метаданными, сравнением (`Comparable`), `equals/hashCode` где нужно.

**Где в коде.**

- `domain/features/research/Researcher.java`, `ResearchPaper.java`.
- Реализации интерфейса — у `Teacher`, `Student`, `ResearchStaff` и т.д.
- Готовые компараторы для сортировки: `domain/sort/ResearchPaperComparators.java` и связанные интерфейсы.

**Демо.** В консоли можно ввести одну статью преподавателю и потом вывести каталог исследователей с сортировкой (по дате / цитатам / страницам).

---

### Фича F — Сценарий «каталог исследователей»

**Что это.** Отдельный класс-сценарий: найти всех пользователей-исследователей, вывести их статьи в нужном порядке, найти «топ» по сумме цитирований или по году и т.д.

**Зачем.** Это слой **application**: доменные классы не захламляются запросами «ко всей базе».

**Где в коде.**

- `application/usecase/ResearcherDirectoryUseCase.java`.
- Зависит только от интерфейса `UserRepository`, а не от конкретной базы — удобно для понимания и тестов на будущее.

---

### Фича G — Расширенный поиск (regex)

**Что это.** Use case принимает **регулярное выражение** и ищет совпадения в email пользователей и в названии курсов.

**Где в коде.**

- `application/usecase/AdvancedSearchUseCase.java`.

**Честно для защиты.** В текущем `ConsoleDemo` этот сценарий **не вызывается из меню**, но класс есть и его можно показать как «готовое расширение» или быстро вызвать из маленького `main` в комментарии — если препод спросит «где поиск».

---

### Фича H — Регистрация студента на курс через use case

**Что это.** Отдельный класс под операцию записи студента на курс (вынесено из консоли).

**Где в коде.**

- `application/usecase/RegisterStudentForCourseUseCase.java`.
- В демо запись идёт напрямую через `student.registerForCourse` — можно сказать: «можно централизовать через use case, сейчас демо упрощено».

---

### Фича I — Исключения предметной области

**Что это.** Не всегда достаточно `IllegalArgumentException`. Есть свои ошибки под правила домена.

**Примеры.**

- Участник проекта должен быть исследователем — `domain/exception/NonResearcherParticipantException.java`.
- Качество научрука для 4 курса — `SupervisorQualificationException.java`.

**Фраза.**

> Так мы отделяем «ошибку бизнес-правила» от случайных программных сбоев и можем по-разному обрабатывать наверху.

---

### Фича J — Хранилище и сохранение в файл

**Что это.**

- В памяти живёт **singleton** `UniversityDatabase`: списки пользователей, курсов, логов.
- Реализует интерфейсы `UserRepository` и `CourseRepository` из домена.
- Умеет **`saveData` / `loadData`** через Java **serialization** (`ObjectOutputStream` / `ObjectInputStream`) в `data/university-state.ser`.

**Зачем.** Показать, что домен описан интерфейсами, а конкретное хранилище можно заменить (БД) без переписывания сценариев.

**Нюанс для умного вопроса.** Если переименовывали пакеты, старый файл содержит старые имена классов — для этого в загрузчике есть **подмена имён классов** (`MigratingObjectInputStream` в `UniversityDatabase`), чтобы старые дампы не ломались сразу.

**Где в коде.**

- Интерфейсы: `domain/repository/UserRepository.java`, `CourseRepository.java`.
- Реализация: `infrastructure/persistence/UniversityDatabase.java`.

---

### Фича K — Прочее доменное (кратко)

- **`ResearchProject`** — проект с участниками-исследователями; добавление участника с проверкой типа.
- **`Student4thYear`** — надстройка над студентом с правилом для научного руководителя (исключение по порогу h-index).
- **`RecommendationLetter`, `StartupProject`** — дополнительные сущности рядом с пользователями (можно упомянуть как расширение модели).
- **`Log`** — запись действий для административного контекста (хранится в базе как часть состояния).

Не расписывайте всё голосом — держите в голове «есть ещё проект 4 курса и стартапы».

## 5. Пошаговый сценарий демонстрации (как в кино)

Делайте **ровно по экрану**, коротко комментируя:

1. Запуск `ConsoleDemo`.
2. Загрузка дампа — **y** или **n** (если были проблемы со старым файлом — **n**).
3. Ввод данных **преподавателя** (включая h-index).
4. Ввод данных **студента**.
5. Ввод **курса**, добавление преподавателя на курс (автоматически в коде).
6. Опционально — статья преподавателю.
7. Сохранение пользователей и курсов в базу (в коде уже `db.add(...)`).
8. Проверка **пароля** преподавателя (`authenticate`).
9. Запись студента на курс → кредиты.
10. Выбор сортировки статей → вывод каталога исследователей → топ по цитированию.
11. Сохранение в файл — **y**.

Если что-то упало — спокойно: «покажем на коде».

## 6. ООП: что произнести, если спросят «где у вас ООП»

| Идея | Где видно |
|------|-----------|
| **Инкапсуляция** | Поля сущностей, методы вместо свободной правки извне; проверки в `registerForCourse`, в исключениях. |
| **Наследование** | `User` → `Employee` → `Teacher`; `Student` и т.д. |
| **Полиморфизм** | `Researcher` реализуют разные классы; `UserRepository` возвращает `User`, реальный тип разный. |
| **Абстракция** | Интерфейсы репозиториев, `Researcher`, `Observer`, `Subject`. |
| **Паттерны** | Factory (`UserFactory`), Singleton (`UniversityDatabase`), Observer (подписка на курс), Repository (интерфейсы в домене). |

## 7. Типичные вопросы и короткие ответы

**Почему singleton для базы?**  
Чтобы был один источник правды по пользователям и курсам в демо и простая загрузка/сохранение.

**Почему не Spring / не БД?**  
Учебный проект: акцент на ООП и слоях; хранилище легко заменить, интерфейсы уже есть.

**Зачем `UniversityApp`?**  
Чтобы консоль не собирала зависимости вручную — одна точка для базы и use case.

**Что будет при параллельной работе?**  
Текущая реализация для одного процесса; для многопоточности нужны другие коллекции/транзакции — это следующий уровень.

**Почему сериализация, а не JSON?**  
Быстро сохранить граф объектов как есть; минус — хрупкость при смене имён классов (мы частично компенсируем миграцией при загрузке).
